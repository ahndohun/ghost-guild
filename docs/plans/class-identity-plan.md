# 클래스 개성 복원 계획 (Class Identity Plan)

보드 리뷰(2026-07-10 심야): "모든 캐릭터가 사실상 같은 캐릭터로 보인다. 엘프에 궁수가 없고, 클레릭과 팔라딘이 외형 구분이 안 되고, 전투방식·스탯·이펙트 어디에도 차별점이 없다. 클래스별 기대 판타지가 게임에 구현되지 않았다."

실관찰(localhost:5173)·에셋 대조·심 데이터 정량 분석·장르 리서치로 진단을 확정했다. **결론: 데이터 층에는 차별화가 있으나(110노드·시그니처 규칙·기질), 그 전부가 화면에 보이지 않는 채널(수치 모디파이어)에 들어 있다. 보이는 채널(시작 무기·스프라이트·이펙트·애니메이션)은 수렴해 있다.**

## 1. 진단 — 왜 다 똑같아 보이나 (증거 포함)

**A. 첫인상 채널인 시작 무기가 6/11 중복.** swordSweep 시작 = fighter·knight·berserker·dwarf·paladin·elf 6클래스, fireBolt = mage·warlock 2클래스 (`src/sim/data.ts` classDefinitions). 플레이어가 클래스 인상을 형성하는 첫 60초 동안 화면에 보이는 것이 동일한 금색 호(arc) 하나다. 레퍼런스 공식(VS·HoT)은 정반대: **캐릭터 = 시작 공격의 공간 패턴**.

**B. 시그니처 규칙이 전부 '보이지 않는 숫자'.** warlock 흡혈 8%, thief 크리 20%×2, dwarf 히트박스 −25%, knight 접촉 −15%, paladin +2HP/5s, berserker kill-heal/fatigue — 모두 수치 모디파이어로만 존재하고 시각 표현이 0이다. 렌더 계층에 이 규칙들을 그리는 코드가 없다 (`src/render/` 전수 확인).

**C. 스프라이트 3중 문제.** ① 방향별 정지 1프레임뿐 — 공격·걷기 애니메이션이 없어 모든 클래스가 "미끄러지는 인형"으로 동일하게 움직인다. ② 팔레트·실루엣 수렴 — paladin과 priest가 둘 다 금백 중장갑(priest는 스펙상 로브 힐 전문가 — DESIGN §10 "Asset = spec" 규칙 위반), fighter와 thief가 회색 검사로 중복. ③ elf가 검+구슬 기사 실루엣 — 대중이 기대하는 엘프 판타지(궁수)와 정면 충돌, 활이 게임 전체에 존재하지 않음.

**D. 행동(기질) 차이가 관전으로 식별 불가.** guardian의 늦은 도주 vs duelist의 카이팅 밴드는 파라미터 차이일 뿐 텔레그래프가 없다. "코칭이 맞았나"를 확인하는 게 코어 루프인데 그 확인 수단이 없다.

**E. 이펙트 질량 부족.** 18종 무기 이펙트가 코드상 전부 구분되지만(`src/render/weapons.ts` weaponVisualProfiles) 1–8px 캔버스 선(stroke) 기반이라 960×540 전장 뷰에서 존재감이 없다. 적 밀도도 낮아 화면 대부분이 빈 바닥. VS의 "무기가 화면을 삼키는" 후반 판타지와 반대.

**F. 레벨업 피드백이 사운드뿐.** 클래스 어피니티 가중치로 클래스다운 스킬을 뽑아주는 시스템이 이미 있는데(affinityWeapons) 무엇을 배웠는지 화면 고지가 없다. LEVEL_UP_PAUSE_TICKS=36으로 일시정지까지 이미 존재 — 배너만 없다.

## 3. 처방 — 페이즈별 계획

### P1. 시작 무기 전면 분화 (sim 데이터, ~1.5h) — 축1·2
`classDefinitions`의 startingWeapon/affinityWeapons 재배치. **11클래스 = 11개 서로 다른 오프닝 화면.**

| 클래스 | 시작 무기 (현재→변경) | 오프닝 공간 패턴 |
|---|---|---|
| Fighter | swordSweep (유지, **유일 소유자로**) | 전방 호 근접 |
| Knight | swordSweep → **shieldBash** | 전방 원뿔 넉백 |
| Berserker | swordSweep → **whirlwindAxe** | 자기 중심 회전 AoE |
| Dwarf | swordSweep → **earthShatter** | 지면 균열 원형 |
| Paladin | swordSweep → **holySmash** | 전방 강타+자힐 링 |
| Mage | fireBolt (유지, 유일) | 직선 화염 투사체 |
| Warlock | fireBolt → **lifeDrain** | 크림슨 흡혈 빔 |
| Priest | holyBolt (유지) + affinityWeapons 첫 픽 radiantBurst 가중 | 십자 볼트→확장 광휘 |
| Elf | swordSweep+fireBolt → **swordSweep+magicArrow** (표기 "Elven Arrow") | 검 호 + 유도 화살 병행 |
| Thief | throwingAxe → **shadowDaggers** (throwingAxe는 어피니티로) | 3단 단검 부채 |
| Monk | garlicAura (유지, 유일) | 지속 오라 |

- 골든 테스트는 의도 변경으로 깨진다 — 커밋 메시지에 원인 명시 후 갱신(선례 `ee2e47f`).
- 로드아웃·고스트 호환: 스키마 불변(무기는 classId에서 유도). 서버 무변경.

### P1.5. 기계적 개성 — 전투 리듬 + 시그니처 모먼트 (sim 변경, ~3h) — 축3
보드 재강조(2026-07-11): "스프라이트만 다른 똑같은 캐릭터처럼 느껴진다." 시각 교체(P2·P3)만으로는 이 지적을 못 넘는다. **스프라이트를 전부 같은 것으로 바꿔도 행동만으로 클래스가 식별되는 수준**이 목표다. 두 레버:

**(a) 행동 파라미터 스프레드 대확장** (기질 프리셋 튜닝 — 저비용·고가시성)
현 기질 파생은 파라미터 차이가 미세해 관전으로 안 읽힌다. 극단으로 벌린다: Mage 카이팅 밴드를 최장(≈200px, 정지-포격-재이동의 포격 리듬), Knight는 절대 holdGround(밀집 한복판으로 걸어 들어가는 유일한 클래스), Thief는 전리품 우회가 항상 이동 우선순위를 이기게, Priest는 최외곽 순회, Berserker/Monk/Dwarf는 combatLock 반경 확대(적만 보면 돌진). 관전 화면에서 "발 위치"만으로 클래스가 갈리는 게 목표 — HoT의 스탯 패키지 원칙("같은 맵에서도 발 위치가 다르다").

**(b) 시그니처 모먼트 — 수동 %를 이벤트로 승격** (기존 PerkEffect trigger 스키마 재사용: hpBelow/onKill/onEliteKill)
전투를 끊는 "그 클래스만의 순간"을 만든다. 발동 시 P4 배너+전용 FX로 반드시 고지. 우선 6종(임팩트 순):

| 클래스 | 모먼트 | 발동 | 효과(초안 수치) | 화면 |
|---|---|---|---|---|
| Berserker | **Blood Rage** | HP<50% 최초 | 10초 공속+50%·이속+30% | 붉은 오라 점화+비네트, fatigue 시 오라 소등 |
| Paladin | **Judgment** | HP<25%, 런당 1회 | radiantBurst 대발동+HP 30% 회복 | 금색 기둥+링 파동 |
| Knight | **Shield Wall** | 반경 100px 적 5+ | 3초 피해 50% 감소+넉백 펄스 | 강철 링+방패 문양 |
| Warlock | **Dark Pact** | HP<40% | 흡혈 8%→16% (전투 종료까지) | 크림슨 스트림 다발화 |
| Thief | **Shadowstep** | 포위 감지(3방향+) | 반대편 급가속 이탈(쿨 10s) | 잔상 대시+동전 흩날림 |
| Mage | **Arcane Surge** | 레벨업 직후 | 3초 쿨다운 절반 | 청색 룬 회오리 |

나머지 5종(Fighter=흔들리지 않는 완벽 리듬 그 자체가 정체성, Dwarf=연타 12히트 시 무료 earthShatter, Priest=주기 Sanctuary 링, Elf=근접 접근 시 Elven Grace 스텝, Monk=무기 만렙 각성 연출)은 2순위 — 시간 시 추가.
- 전부 seeded 결정론 내에서(트리거는 상태 조건이라 RNG 불요), 골든 갱신 필수. 밸런스 영향은 property 테스트(비전투 정지 방지·클래스별 상이 결과)로 가드.
- 아이템 유니크가 이 모먼트를 변조하면(예: Berserker 유니크 "Bloodhowl Axe"가 Rage 지속 2배) 아이템 시스템과 자연 결합.

### P2. 보이지 않는 규칙 전부에 시각 채널 부여 (render 전용, ~2.5h) — 축4
sim 불변·렌더만 수정이므로 결정론 무風險. `src/render/effects.ts`에 클래스 시그니처 FX 추가:

| 규칙 (invisible) | 시각 채널 (신규) |
|---|---|
| Warlock 흡혈 8% | 피해 시 적→자신 크림슨 파티클 스트림 + 회복 틱 초록 플래시 |
| Berserker kill-heal / 90s fatigue | 처치 순간 붉은 흡수 번쩍 + 상시 분노 오라(fatigue 진입 시 오라 소등 — 약점도 보여준다) |
| Thief 크리 20%×2 | 크리 데미지 숫자 대형+금색, "CRIT!" 스파크 |
| Dwarf 히트박스 −25% | 스프라이트 드로 스케일 실제 축소(hitbox와 정합) + 빠른 케이던스 잔상 |
| Paladin +2HP/5s | 5초 주기 금색 힐 링 파동 |
| Knight 접촉 −15% | 접촉 피격 시 방패 스파크(피해 감소가 보인다) |
| Monk 무기 진화 Lv | 레벨 구간별 오라 색 진화 |
| Priest 힐 특화 | 힐 발동 광휘 + 잔광 |
| 기질(행동) | 텔레그래프: guardian 발밑 방패 문양 / hoarder 전리품 방향 반짝 시선선 / duelist 카이팅 밴드 흐릿한 링 / berserker 돌진 시 붉은 궤적 |

### P3. 스프라이트 교정 3종 (PixelLab 재생성, 병렬 큐 ~1.5h) — 축4
아트 디렉션 잠금(low top-down 32px v3) 유지, 실루엣·팔레트 분리가 목표:
1. **Priest**: 갑옷 제거 → 흰/미색 로브 + 성표 + 두건 or 삭발. paladin(금 갑옷+헤일로)과 즉시 구분.
2. **Elf**: **활 + 화살통** 궁수 실루엣, 녹색 경장. P1의 Elven Arrow와 정합.
3. **Berserker**: 웃통/모피 + 양손 도끼 + 광기 표현(현재는 귀족풍 모피 코트).
파이프라인: 생성→다운로드→즉시 커밋→스프라이트 디렉토리 교체(등록 구조 불변).
(2순위 후보: fighter/thief 회색 수렴 — 시간 남으면 thief에 보라 스카프 등 포인트 컬러 재생성.)

### P4. 레벨업·시그니처 코멘터리 (UI+render, ~1h) — 피드백 루프
- 레벨업 일시정지(이미 존재) 중 DQ풍 배너: "팔라딘은 홀리 스매시를 익혔다!" — 클래스 어피니티 픽이 클래스 서사가 된다.
- 시그니처 첫 발동 1회성 미니 배너(런당): "도둑의 눈이 번뜩인다 — 치명타!" 등. 게임이 스스로 클래스 규칙을 설명.
- DOM이 아닌 캔버스 오버레이로(전투 층에서 재생), testid 불필요 — E2E 표면 불변.

### P5. 질량·밀도 상향 (선택, 시간 남으면)
- 렌더 전용 안전판: 이펙트 두께·크기 1.5~2배, 히어로/적 드로 스케일 상향(히트박스 불변 주의 — 겹침 가독성 확인).
- 밸런스 위험판: 웨이브 밀도 상향은 sim 변경 — 마감 직전엔 금지.

## 4. 수용 기준 (블라인드 테스트 — 2단계)

1. **[1단계·시각] 같은 시드로 클래스 3종(예: knight/warlock/thief) 각 30초 녹화 → 라벨 가리고 어느 클래스인지 식별.**
1-2. **[2단계·기계적 — 보드 재강조 기준] 스프라이트를 전부 동일한 더미로 강제 교체한 상태(렌더 디버그 플래그)에서도, 발 위치·전투 리듬·시그니처 모먼트만으로 클래스 식별이 가능해야 한다.** "스프라이트만 다른 같은 캐릭터"의 정확한 반증 테스트다. knight(밀집 돌파)/mage(원거리 포격 리듬)/thief(전리품 우회+이탈 대시) 3종부터 통과시킬 것.
2. 클래스 판타지 게이트(docs/roster-v3.md §fantasy) 11문장 대비: 각 문장이 "화면에서" 확인돼야 한다 (카드 텍스트 말고).
3. paladin vs priest 스프라이트를 나란히 놓고 3초 안에 구분.
4. 게이트: typecheck·vitest(골든 갱신 포함)·build → 배포 → tsloop → LOOP 기록.
5. 결정론: 같은 시드 2회 실행 동일 결과(기존 property 테스트), 고스트 리플레이 무결.

## 5. 실행 노트
